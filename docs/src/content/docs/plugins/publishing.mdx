The EmDash Marketplace lets site owners install plugins directly from the admin dashboard. Publishing your plugin makes it available to every EmDash site.

## The `emdash publish` Command

The simplest path from source to marketplace is a single command:

```bash
emdash plugin publish --build
```

This bundles your plugin, validates it, and uploads it in one step. You can also separate the steps:

```bash
# Build the bundle
emdash plugin bundle

# Inspect it
tar tzf dist/my-plugin-1.0.0.tar.gz

# Publish
emdash plugin publish
```

### Authentication

The first time you publish, the CLI authenticates via GitHub:

<Steps>
1. The CLI opens your browser to GitHub's device authorization page
2. You enter the code displayed in your terminal
3. GitHub issues an access token
4. The CLI exchanges it for a marketplace JWT (stored in `~/.config/emdash/auth.json`)
</Steps>

The token lasts 30 days. Manage authentication separately with:

```bash
emdash plugin login    # Authenticate without publishing
emdash plugin logout   # Clear stored credentials
```

## The `replaces` Field

If your plugin replaces one or more WordPress plugins, declare that in the descriptor:

```typescript title="src/index.ts"
export function seoPlugin(): PluginDescriptor {
	return {
		id: "seo",
		version: "0.1.0",
		replaces: ["yoast-seo", "rank-math-seo", "all-in-one-seo-pack"],
		// ...
	};
}
```

The `replaces` array lists WordPress plugin slugs (the slug from the WordPress plugin directory URL, e.g., `wordpress.org/plugins/yoast-seo/`). This serves two purposes:

1. **Migration UI.** When a site migrates from WordPress, EmDash suggests your plugin as a replacement for the WordPress plugins it replaces.
2. **Marketplace search.** Users searching for "yoast" or "rank math" find your plugin.

## Bundle Format

The `emdash plugin bundle` command produces a `.tar.gz` tarball containing:

| File | Required | Description |
| --- | --- | --- |
| `manifest.json` | Yes | Plugin metadata extracted from `definePlugin()` |
| `backend.js` | No | Bundled sandbox code (self-contained ES module) |
| `admin.js` | No | Bundled admin UI code |
| `README.md` | No | Plugin documentation |
| `icon.png` | No | Plugin icon (256x256 PNG) |
| `screenshots/` | No | Up to 5 screenshots (PNG/JPEG, max 1920x1080) |

The `manifest.json` is generated automatically from your code. It contains the plugin ID, version, capabilities, hook names, and route names -- but no executable code.

### Bundle Validation

The bundler checks:

- **Size limit** -- Total bundle must be under 5MB.
- **No Node.js built-ins** -- `backend.js` cannot import `fs`, `path`, `child_process`, etc. Sandbox code runs in a V8 isolate, not Node.js.
- **Icon dimensions** -- `icon.png` should be 256x256 (warns if wrong).
- **Screenshot limits** -- Max 5 screenshots, max 1920x1080.

<Aside type="tip">
	If your backend code imports a Node.js built-in, the bundle will fail. Replace Node.js APIs with
	Web APIs (e.g., use `ctx.http.fetch` instead of `node:http`) or move the logic to a trusted
	plugin that is not published to the marketplace.
</Aside>

## Marketplace Security Audit

Every published version goes through an automated, AI-powered security scan before it appears in the marketplace. The audit analyzes your `backend.js` and `admin.js` for:

- **Data exfiltration** -- Attempts to send user data to undeclared hosts
- **Credential harvesting** -- Using settings fields to collect API keys and send them externally
- **Obfuscated code** -- Minification is fine; deliberate obfuscation (eval, Function constructor, base64-encoded logic) is flagged
- **Resource abuse** -- Crypto mining, excessive computation, background network activity
- **Capability misuse** -- Requesting capabilities the code does not appear to use, or using APIs beyond declared capabilities

The audit produces a verdict:

| Verdict | Meaning | Effect |
| --- | --- | --- |
| **pass** | No issues found | Plugin is published immediately |
| **warn** | Minor concerns | Published with warnings visible on the listing |
| **fail** | Security issues detected | Publication blocked until issues are resolved |

The verdict and summary are displayed on the plugin's marketplace listing page, so users can make informed installation decisions.

## Version Management

Each published version must have a **higher semver** than the last. You cannot overwrite or republish an existing version.

```bash
# In definePlugin() and package.json, bump the version:
# 0.1.0 -> 0.2.0

# Then publish
emdash plugin publish --build
```

The version in your `definePlugin()` call and your `package.json` must match. The bundler checks this.

### First-Time Registration

If your plugin ID does not exist in the marketplace yet, `emdash plugin publish` registers it automatically before uploading the first version. No separate registration step is needed.

## CLI Reference

### `emdash plugin bundle`

```bash
emdash plugin bundle [--dir <path>] [--outDir <path>]
```

| Flag | Default | Description |
| --- | --- | --- |
| `--dir` | Current directory | Plugin source directory |
| `--outDir`, `-o` | `dist` | Output directory for the tarball |

### `emdash plugin publish`

```bash
emdash plugin publish [--tarball <path>] [--build] [--dir <path>] [--registry <url>]
```

| Flag | Default | Description |
| --- | --- | --- |
| `--tarball` | Latest `.tar.gz` in `dist/` | Explicit tarball path |
| `--build` | `false` | Run `emdash plugin bundle` first |
| `--dir` | Current directory | Plugin directory (used with `--build`) |
| `--registry` | `https://marketplace.emdashcms.com` | Marketplace URL |

## Complete Publish Workflow

```bash
# 1. Develop and test your plugin locally
# 2. Bump version in definePlugin() and package.json
# 3. Build, validate, and publish in one step
emdash plugin publish --build

# Or, if you want to inspect the bundle first:
emdash plugin bundle
tar tzf dist/my-plugin-1.1.0.tar.gz
emdash plugin publish
```

After publishing, your plugin appears in the EmDash Marketplace within minutes. Site owners can install it from the admin dashboard under Plugins > Marketplace.
